## 04_名片夾 - 測試案例文件

### 一、需求結構分析表（背景分析）

| 模組 | 核心功能 | 輸入規則 | 相依關係 |
|------|----------|----------|----------|
| 全域搜尋 | 依姓名 / 職稱 / 公司名稱關鍵字搜尋名片，支援邊打邊搜與 debounce | 關鍵字可為任意字串，空值時視為「不套用搜尋條件」，輸入變更 300ms 內無再輸入才觸發搜尋 | 相依：名片列表顯示、名片總張數、空狀態顯示、分類標籤狀態（搜尋時隱藏） |
| 分類標籤 | 在「我的最愛 / AiCard名片 / 聯絡人」三種分類間切換列表內容，並避免快速切換頻繁呼叫 API | 三個標籤其一為選取狀態，點擊後 300ms 內未再切換才觸發 API | 相依：名片列表顯示、名片總張數、空狀態顯示、全域搜尋（搜尋時需隱藏分類標籤） |
| 名片列表（分頁 & 無限滾動） | 依目前條件顯示名片列表，每頁 20 張，向下滾動載入下一頁 | 每次載入固定 20 筆；當不足 20 筆時代表已到最後一頁；需處理重複載入與空清單 | 相依：全域搜尋、分類標籤、名片預覽區塊、名片總張數、名片詳細頁 |
| 名片預覽樣式 | 依名片類型顯示不同樣式（AiCard名片–藍 / 聯絡人–灰），顯示頭圖、姓名、職稱、公司名稱 | 頭圖可能為空；姓名、職稱、公司名稱可能存在缺漏，需要定義顯示邏輯 | 相依：Figma 設計規範、收藏狀態、聯絡功能按鈕區 |
| 收藏功能 | 透過星星 icon 切換名片是否在「我的最愛」，實心=已收藏、空心=未收藏，操作需有 toast 回饋 | 點擊後需立即更新 icon 狀態與伺服器資料，避免重複點擊導致狀態錯亂 | 相依：我的最愛標籤資料、名片列表即時刷新或局部更新、toast 系統 |
| 聯絡功能 - LINE | LINE 按鈕點擊後前往 LINE 聊天室，若未填寫 LINE 資訊則顯示 disabled icon 無法點擊 | 需檢查 LINE ID / 連結是否存在；disabled 狀態不可觸發行為 | 相依：裝置已安裝 LINE、系統 URL Scheme、名片資料欄位 |
| 聯絡功能 - 手機 | 手機按鈕點擊後呼叫手機原生撥號功能 | 需檢查電話格式與存在與否，避免空值仍可點擊 | 相依：系統撥號器、名片資料欄位 |
| 聯絡功能 - Email | Email 按鈕點擊後呼叫手機原生郵件功能，未填寫時顯示 disabled icon | 需檢查 Email 是否存在；disabled 狀態不可觸發行為 | 相依：系統預設郵件 App、名片資料欄位 |
| 名片總張數顯示 | 顯示目前列表中的名片總數，格式「X 張名片」，無結果時顯示「0 張名片」 | 需與實際列表數量同步更新，搜尋或切換分類後應重新計算 | 相依：全域搜尋、分類標籤、名片列表資料來源 |
| 空狀態顯示 | 當無任何名片結果時，顯示空狀態畫面與「0 張名片」 | 需定義空狀態文案與圖示；搜尋條件與分類不同情境皆需顯示正確 | 相依：全域搜尋、分類標籤、名片列表資料來源 |

### 二、輸入空間與邊界表（背景分析）

| 欄位/項目 | 合法範圍 | 邊界值 | 異常情境 |
|-----------|----------|--------|----------|
| 搜尋關鍵字 | 任意可輸入字串（含中英數、符號） | 空字串、1 字元、255 字元、超過後端限制長度（⚠️ 待釐清：最大長度） | 輸入極長字串、輸入特殊符號、連續快速輸入與刪除、只輸入空白 |
| 分類標籤選取 | `我的最愛` / `AiCard名片` / `聯絡人` 其一 | 初次進入預設標籤（⚠️ 待釐清：預設是哪一個）、快速來回點擊兩個不同標籤 | 連點同一標籤、快速切換多個標籤、API 逾時或失敗 |
| 名片列表分頁 | 每頁 20 筆，最後一頁可 < 20 筆 | 剛好 20 筆、21 筆（2 頁）、39/40/41 筆、0 筆 | API 回傳重複名單、分頁參數錯誤、滾動事件多次觸發 |
| 名片類型 | AiCard名片 / 聯絡人 | 無 | 類型值異常或缺失（⚠️ 待釐清：fallback 樣式） |
| 頭圖 | 有設定圖片 / 無圖片 | 有圖、高解析圖、極小圖、無圖 | 圖片載入失敗、連線中斷 |
| 姓名 | 非空字串 | 1 字元、長姓名（中/英）、含特殊符號 | 空值、超過顯示寬度需換行或截斷 |
| 職稱 | 可為空或非空字串 | 1 字元、長職稱、多行顯示 | 職稱空值顯示方式（⚠️ 待釐清） |
| 公司名稱 | 可為空或非空字串 | 1 字元、長公司名、多行顯示 | 公司名稱空值顯示方式（⚠️ 待釐清） |
| 收藏狀態 | 已收藏 / 未收藏 | 初始為已收藏、初始為未收藏 | 快速連點星星、API 失敗、同步至「我的最愛」列表延遲 |
| LINE 資訊 | 有 LINE ID 或連結 / 未填寫 | 正確 ID、錯誤格式 ID | 未填寫卻顯示可點、已填寫卻顯示 disabled、點擊時未安裝 LINE（⚠️ 待釐清） |
| 手機號碼 | 合法手機號碼格式（⚠️ 待釐清：格式規則） | 10 碼正確、9/11 碼錯誤 | 空值但按鈕可點、錯誤格式仍可撥號 |
| Email | 合法 email 格式 | 正確 email、長 email（多個子網域）、大寫字母 | 空值但未 disabled、格式錯誤仍可點擊觸發 |
| 名片總張數 | 非負整數 | 0、1、20、21、99、100、999+（⚠️ 待釐清：顯示上限格式） | 顯示數量與實際列表不一致、更新延遲 |

### 三、風險問題表（背景分析）

| 問題 | 有則補案例 |
|------|-----------|
| 這個動作需要對方配合嗎？對方不在線、拒絕、或中途改變狀態會怎樣？ | 名片夾主要為瀏覽與收藏，與對方線上狀態關聯較低；但聯絡功能（LINE / 電話 / Email）涉及對方是否可被聯繫，偏屬外部系統風險，不在本模組核心範圍，僅需確認跳轉是否成功 |
| 使用者可以中途離開嗎？關掉頁面、斷網、切換 app 會怎樣？ | 需補：滾動載入中途離開 / 重新回來是否維持捲動位置（⚠️ 待釐清）、收藏操作後尚未同步完畢即離開頁面時的狀態一致性 |
| 這筆資料對方可以刪除或修改嗎？ | 若對方刪除名片或變更資料，名片夾是否即時更新或在下次開啟時更新（⚠️ 待釐清）；需補至少一則重新整理資料的案例 |
| 這個動作會被重複觸發嗎？快速點擊、多人同時操作會怎樣？ | 需補：搜尋輸入快速變更、分類標籤快速切換、名片無限滾動快速到底、收藏星星連點、聯絡按鈕連點 等多則例外案例 |
| 流程有多個步驟，每個步驟失敗各自會怎樣？ | 搜尋：輸入 → debounce → 呼叫 API → 顯示結果；每一步 API 逾時/失敗都需補案例；列表載入更多：捲動偵測 → 呼叫下一頁 API → append 結果；收藏：點擊 → 本地 UI 更新 → API 回應；各步驟失敗需定義預設行為並補案例 |
| 有顯示時間、數量、或狀態的 UI 元件嗎？ | 名片總張數為數量顯示元件，需補：0、1、小於 20、恰好多頁分界、大量名片時是否截斷；收藏狀態 icon、聯絡按鈕 enabled/disabled 皆需逐一 case |

### 五、⚠️ 待釐清事項彙總（供 PM/RD 回覆）

1. **搜尋空白行為**（TC-04-SEARCH-013）  
   - 僅輸入空白字元時，是否視為「無搜尋條件」並還原完整列表？  
2. **搜尋特殊符號支援度**（TC-04-SEARCH-014）  
   - 特殊符號是否允許作為關鍵字？是否需要跳脫或特別處理？  
3. **搜尋關鍵字長度上限 N**（TC-04-SEARCH-009, 010）  
   - 後端允許的最大長度與前端是否需同步限制？  
4. **初次進入名片夾預設分類**（TC-04-TAB-009）  
   - 預設應顯示哪一個分類標籤？是否記憶使用者上次選擇？  
5. **頭圖 / 職稱 / 公司名稱空值顯示策略**（TC-04-CARD-003, 005, 006）  
   - 頭圖沒有時顯示預設圖還是留空？  
   - 職稱 / 公司名稱為空時，是整行隱藏還是顯示「—」之類預設文案？  
6. **大量名片與總張數上限顯示規則**（TC-04-LIST-007, TC-04-COUNT-006）  
   - 當名片超過一定數量（例如 999）時，是否顯示「999+ 張名片」？  
7. **收藏失敗與快速連點行為**（TC-04-FAV-004, 005, 006）  
   - 收藏 API 失敗時，UI 是否需要回復原狀？是否有 retry 機制？  
   - 快速連點收藏時，最終以哪一次為準？後端是否保證最終一致性？  
8. **聯絡功能錯誤情境**（TC-04-CONTACT-008, 009）  
   - 裝置未安裝 LINE 時應顯示何種提示？  
   - 電話格式錯誤是否要在前端攔截並提示，而非直接送給 OS？  
9. **返回名片夾時是否保留捲動位置**（TC-04-DETAIL-001）  
   - 從名片詳細頁返回時，是否需回到原本滾動位置？  
10. **收藏後列表是否立即重新排序或重載**（TC-04-FAV-007）  
    - 收藏狀態改變後是否影響目前列表排序？是否需要即時重新載入？

### 六、Mock / 測試資料建議

- **帳號與裝置**
  - 測試帳號 A：一般使用者，名片數量中等（約 50 筆），包含 AiCard 與聯絡人
  - 測試帳號 B：名片極少（0~1 筆），用於空狀態與邊界測試
  - 測試帳號 C：名片極多（200+ 筆），用於無限滾動與效能觀察
  - 測試裝置需至少包含 iOS 與 Android 各一款，且安裝/未安裝 LINE 的組合

- **名片資料組合**
  - 至少 1 組：完整資料（頭圖＋姓名＋職稱＋公司）
  - 至少 1 組：無頭圖
  - 至少 1 組：長姓名、長職稱、長公司名
  - 多組：僅姓名＋公司 / 僅姓名＋職稱 等欄位缺漏組合
  - 各分類（我的最愛 / AiCard名片 / 聯絡人）需有資料覆蓋

- **聯絡資訊組合**
  - LINE / 手機 / Email 皆有填資料
  - 僅填 LINE
  - 僅填手機
  - 僅填 Email
  - 三者皆未填
  - 錯誤格式範例：電話少於 10 碼、Email 無 `@`、LINE ID 含非法字元（視規則而定）
